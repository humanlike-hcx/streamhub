# StreamHub 后续开发路线图

> 本路线图基于当前项目进度整理，只规划后续开发方向，不改变现有业务代码。

## 当前进度

当前 StreamHub 已完成视频网站后端 MVP 核心链路：

- Spring Boot 3.5 + Java 17 基础工程
- 用户注册登录
- JWT + Spring Security 无状态认证
- MinIO 原始视频对象存储
- RocketMQ 异步转码消息
- FFmpeg 生成 HLS
- HLS 后端代理播放
- 视频详情接口
- 公开视频列表接口
- 我的视频列表接口

当前架构约束：

- 转码链路使用 RocketMQ，不再使用 RabbitMQ。
- 不使用 Canal。
- 弹幕模块使用 Netty WebSocket。
- Controller 只做参数接收、参数校验和结果返回。
- 业务逻辑放在 Service。
- 数据库访问放在 Mapper。
- 视频文件不进入 MySQL，MySQL 只保存元数据和 object key。

## 阶段 1：转码链路可靠性增强

### 目标

把当前可跑通的转码链路从 MVP 版增强为更可靠的异步任务系统，解决重复消息、并发 Consumer、失败重试、消息发送失败和对象存储残留问题。

### 涉及技术

- RocketMQ
- MySQL 条件更新
- MyBatis-Plus 自定义 SQL
- Spring 事务
- Publisher Confirm
- 定时任务
- MinIO 对象清理

### 重点任务

- 实现转码任务原子抢占：
  - 使用 `UPDATE transcode_tasks SET status='PROCESSING' WHERE id=? AND status='WAITING'`。
  - 通过影响行数判断当前 Consumer 是否抢占成功。
- 实现消费幂等：
  - 重复消息到达时，如果任务已不是 `WAITING`，直接忽略。
  - 避免同一个视频被重复转码。
- 增加失败重试：
  - 新增 `max_retry_count`。
  - 使用 `retry_count` 和 `next_retry_at` 控制重试次数和重试时间。
- 增加 PROCESSING 超时恢复：
  - 新增 `locked_by`、`locked_at`。
  - 定时扫描长时间卡在 `PROCESSING` 的任务，重新置为 `WAITING` 或进入失败重试。
- 增加 RocketMQ 发送结果校验和失败补偿：
  - 解决数据库事务已提交但 MQ 消息发送失败的问题。
  - 可先做 confirm 日志和补偿扫描，后续再考虑 outbox 表。
- 增加 MinIO 清理：
  - 转码失败时清理本次生成的 HLS 临时目录。
  - 删除视频时清理 original、hls、cover 对象。

### 预期产出

- 更可靠的 `TranscodeTaskService`
- 原子抢占方法，例如 `tryClaimTask(...)`
- 重试和超时恢复定时任务
- RocketMQ producer/consumer 配置
- MinIO `deleteObject(...)`、`deletePrefix(...)`
- 更新后的 `docs/sql/schema.sql`

### 面试讲解点

- RocketMQ 可能重复投递，业务幂等不能只依赖 MQ。
- 真正的“只处理一次”靠数据库任务表的原子状态更新实现。
- `afterCommit` 解决事务未提交先发消息的问题，RocketMQ send result 和补偿扫描解决 MQ 发送失败的问题。
- 任务表是异步任务的事实来源，MQ 只是通知。
- 对象存储和 MySQL 不在同一个事务里，所以需要补偿清理。

## 阶段 2：视频元数据增强

### 目标

让视频展示信息更完整，支持真实时长、分辨率和封面，提升视频详情页和列表页质量。

### 涉及技术

- FFmpeg
- ffprobe
- MinIO
- MySQL 元数据字段

### 重点任务

- 使用 `ffprobe` 提取：
  - 视频时长
  - 宽度
  - 高度
  - 编码格式
- 使用 FFmpeg 抽帧生成封面。
- 上传封面到 MinIO。
- 保存封面 object key 或访问 URL。
- 更新 `videos.duration_seconds`。
- 可新增字段：
  - `width`
  - `height`
  - `cover_object_key`

### 预期产出

- 视频详情返回时长、分辨率、封面
- 视频列表展示封面和时长
- 转码成功后自动补齐视频元数据
- 更新后的 `docs/sql/schema.sql`

### 面试讲解点

- 原始视频上传后不能直接假设元数据可信，应由后端通过 `ffprobe` 解析。
- 封面属于对象存储资源，MySQL 保存 object key。
- 视频详情页需要的是业务字段，不应该暴露内部存储路径。

## 阶段 3：互动模块

### 目标

补齐视频平台的基础用户互动能力。

### 涉及技术

- MySQL
- MyBatis-Plus
- Redis 可选
- Spring Validation

### 重点任务

- 点赞：
  - 用户对视频点赞、取消点赞。
  - 防止重复点赞。
- 收藏：
  - 用户收藏、取消收藏。
  - 我的收藏列表。
- 评论：
  - 发表评论。
  - 评论分页。
  - 删除自己的评论。
- 播放量：
  - 播放接口或前端播放事件触发播放统计。
  - 初期可直接写 MySQL，后续改 Redis 异步落库。

### 预期产出

- 点赞接口
- 收藏接口
- 评论接口
- 播放量统计字段
- 对应数据库表和索引

### 面试讲解点

- 点赞和收藏需要唯一约束防止重复。
- 评论是典型分页查询场景。
- 播放量高频写入不适合直接频繁刷 MySQL，后续应引入 Redis 聚合。

## 阶段 4：Redis 热门榜和播放量异步落库

### 目标

用 Redis 承担高频播放量、点赞量和热度统计，降低 MySQL 写压力。

### 涉及技术

- Redis
- ZSet
- String / Hash
- 定时任务
- RocketMQ 可选

### 重点任务

- 播放量先写 Redis。
- 使用 Redis ZSet 维护热门榜。
- 定时将 Redis 中的播放量增量落库到 MySQL。
- 设计热度分：
  - 播放量
  - 点赞数
  - 收藏数
  - 评论数
  - 发布时间衰减

### 预期产出

- 热门视频榜接口
- 播放量异步落库任务
- Redis key 设计文档
- 热度分计算策略

### 面试讲解点

- 高频计数不应每次都写 MySQL。
- Redis ZSet 天然适合排行榜。
- 异步落库需要考虑幂等和数据丢失风险。

## 阶段 5：Netty WebSocket 弹幕

### 目标

实现实时弹幕能力，支持用户在视频播放页发送和接收弹幕。

### 涉及技术

- Netty
- WebSocket
- JWT 鉴权
- Redis
- MySQL
- 限流

### 重点任务

- Netty WebSocket 服务。
- 房间管理：
  - 一个视频对应一个弹幕房间。
  - 用户进入播放页时加入房间。
  - 用户离开时退出房间。
- JWT 鉴权：
  - WebSocket 握手阶段校验 token。
  - 解析用户身份。
- 心跳机制：
  - ping/pong。
  - 清理断开的连接。
- 实时弹幕广播：
  - 用户发送弹幕。
  - 服务端广播给同房间用户。
- 历史弹幕：
  - MySQL 保存弹幕。
  - 查询某视频历史弹幕。
- 限流：
  - 单用户发送频率限制。
  - 防刷屏。

### 预期产出

- Netty WebSocket 服务模块
- 弹幕房间管理
- 弹幕发送和广播
- 历史弹幕查询接口
- 限流策略

### 面试讲解点

- 弹幕是实时长连接场景，适合 WebSocket。
- 使用 Netty 可以更细粒度控制连接、心跳、编解码和资源管理。
- JWT 鉴权应在握手阶段完成，避免匿名连接滥用。
- 房间模型可以按 `videoId` 组织连接。

## 阶段 6：分片上传、断点续传、MD5 秒传、Redisson 分布式锁

### 目标

优化大文件上传体验，支持上传中断后续传，以及相同文件秒传。

### 涉及技术

- MinIO 分片上传
- Redis
- Redisson
- MD5 / SHA-256
- MySQL 上传会话表

### 重点任务

- 创建上传会话。
- 分片上传。
- 记录分片状态。
- 断点续传：
  - 查询已上传分片。
  - 只上传缺失分片。
- MD5 秒传：
  - 文件 hash 已存在时，直接复用对象。
- Redisson 分布式锁：
  - 防止同一文件 hash 并发秒传或合并。
  - 防止同一上传会话并发完成。

### 预期产出

- 分片上传接口
- 查询上传进度接口
- 完成上传接口
- 秒传接口
- 上传会话表和分片表

### 面试讲解点

- 大文件上传不能完全依赖一次 multipart。
- 断点续传的关键是上传会话和分片状态。
- 秒传本质是文件内容寻址。
- 分布式锁用于保护合并和 hash 复用的临界区。

## 阶段 7：搜索模块

### 目标

支持按标题、描述、作者等维度检索视频。

### 涉及技术

- ElasticSearch 或 OpenSearch
- MySQL
- RocketMQ 可选

### 重点任务

- 设计视频搜索索引。
- 视频发布后写入搜索索引。
- 视频更新后同步索引。
- 视频删除后删除索引。
- 搜索接口：
  - 关键词搜索
  - 分页
  - 排序
  - 高亮可选

### 预期产出

- 搜索索引 mapping
- 搜索同步 Service
- 搜索接口
- 索引重建脚本或任务

### 面试讲解点

- MySQL `LIKE` 适合简单搜索，不适合复杂全文检索。
- 搜索索引和数据库之间存在最终一致性。
- 索引重建能力是搜索系统的兜底手段。

## 推荐近期开发顺序

近期建议优先做：

1. 转码链路可靠性增强。
2. 视频元数据增强。
3. 点赞、收藏、评论。
4. Redis 热门榜。
5. Netty WebSocket 弹幕。

原因：

- 当前核心链路已经跑通，下一步最能提升项目深度的是可靠性。
- 视频元数据增强能明显提升产品完整度。
- 互动、热门榜、弹幕能让项目更像真实视频平台。
